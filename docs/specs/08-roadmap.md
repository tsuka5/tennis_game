# 08. 実装ロードマップ（Claude Codeへの指示順）

元仕様のフェーズ構成を踏襲しつつ、各フェーズに **完了定義（DoD）** と **対応AC** を付す。
各フェーズは前フェーズのDoDを満たしてから着手する（MUST）。

## フェーズ 0: プロジェクト基盤（前提整備）

- Vite + TypeScript + Phaser 3 のプロジェクト雛形。
- `GameConfig`（[06](./06-data-models.md) 6.6）と論理座標定数を定義。
- HTTPSローカル配信（実機テスト用）と、横画面固定・縦持ち案内の土台。
- **DoD**: スマホ実機で空のコートが横画面表示され、`GameConfig`から寸法が反映される。
- **対応AC**: AC-1.7, AC-6.2

## フェーズ 1: スマホ単体・壁打ちプロトタイプ

- 2Dトップダウンのコート描画（ネット・ライン・自キャラ・ボール）。
- 左画面=動的バーチャルジョイスティック移動、右画面=A/Bボタン。
- Arcade Physics でボール物理＋擬似高さ`height`、A/スマッシュ/ロブ/バウンド/ネット判定。
- 相手側の「壁」反射でラリーが継続。
- **DoD**: オフラインで起動し、移動・各ショット・壁打ちラリーが成立。
- **対応AC**: AC-1.1〜1.6, AC-3.5, AC-4.1（ローカル即時ヒットの土台）, AC-5.1〜5.3（判定ロジック単体）

## フェーズ 2: 画面180度反転の実装

- 論理座標 ⇔ ローカル描画変換（ホスト恒等 / クライアント180°）。
- 入力の反転同期（[02](./02-coordinate-and-camera.md) 2.5）。
- 2つのブラウザ（または1台で2視点）で、片方を反転表示しても位置が正しく対応することをローカル結合（疑似ネット）で確認。
- **DoD**: 反転表示下でも双方「自分が手前」、同一論理状態が破綻なく両画面に描画。逆変換が可逆。
- **対応AC**: AC-2.1〜2.4

## フェーズ 3: WebRTC 1vs1 対戦（コア）

- 最小WebSocketシグナリングサーバ（[01](./01-architecture.md) 1.3）。ルーム作成/参加/SDP・ICE中継。
- URL/QR共有での参加フロー。STUNのみでP2P確立。
- DataChannel上の `Envelope`/メッセージ（[06](./06-data-models.md)）。信頼性の使い分け（[01](./01-architecture.md) 1.5）。
- ボール権限・ハンドオフ・時刻同期・補間（[05](./05-netcode-sync.md)）。
- 人工遅延注入のデバッグ手段を用意（SHOULD）。
- **DoD**: スマホ2台でラグなくラリーが往復し、空振りゼロ・ボール消失/二重化なし。
- **対応AC**: AC-3.1〜3.4, AC-4.1〜4.6

## フェーズ 4: UI・スコア・部屋設定

- 得点演出、ポイント間インターバル、ゲームセット画面、再戦。
- ホスト権威のスコア同期（POINT_REPORT→POINT_RESULT、[05](./05-netcode-sync.md) 5.4.4）。
- ルーム作成画面: モード（シングルス/ダブルス）と先取ポイント数の選択。
- ロビー/参加者表示、切断・ホスト消滅の通知。
- **DoD**: 1試合が最後まで進行し、両端末でスコアが一致、ゲームセット表示、再戦可能。
- **対応AC**: AC-3.4, AC-5.4〜5.6, AC-4.6

## フェーズ 5: ダブルス（2vs2）※拡張

- スター型で最大4人接続、チーム単位のボール権限・代表決定（[05](./05-netcode-sync.md) 5.5）。
- ホスト経由のクライアント間中継。
- **DoD**: 4人接続でダブルスのラリー・得点・ゲームセットが成立。
- **対応AC**: AC-3系（定員4）, AC-4系（中継下）, AC-5系

## フェーズ 6: 仕上げ・非機能

- 実機パフォーマンス調整（目標60fps / 最低30fps）、Wake Lock、ハプティクス。
- 軽量サニティチェック（[05](./05-netcode-sync.md) 5.7）。
- バランス調整（`GameConfig`）。
- **DoD**: 対象実機で目標フレームレート、操作快適性、運用コスト¥0を確認。
- **対応AC**: AC-6.1〜6.3
