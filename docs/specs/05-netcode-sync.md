# 05. P2P同期・ラグ隠蔽・ボール権限委譲

本章は本ゲームの核心。「ラケットに当たったか」を**自分の画面では絶対に空振りしない**よう保証する（仕様6）。

## 5.1 同期モデル概要

- すべての同期は**論理座標**（[02](./02-coordinate-and-camera.md)）で行う（MUST）。
- 状態は2種類に分ける:
  - **高頻度・非信頼**: 各プレイヤーの位置/速度、現在ボール権限ピアが流すボール状態（最新値優先、欠落許容）。
  - **低頻度・信頼**: 権限の移譲（ハンドオフ）、得点、試合状態、サーブ（取りこぼし禁止）。
- ローカル描画は**補間/外挿**でなめらかに見せる（MUST）。

送信レート（既定・調整可）:

| データ | レート |
|---|---|
| PLAYER_STATE（自キャラ） | 20〜30 Hz |
| BALL_STATE（権限保持時のみ） | 20〜30 Hz |
| PING/PONG | 1 Hz |
| イベント系（HANDOFF/POINT/MATCH） | 発生時のみ（reliable） |

## 5.2 時刻同期（Clock Sync）

- **MUST**: PING/PONGでRTTと相手とのクロックオフセットを推定し、受信状態の時刻整合に用いる。
- 各メッセージは送信側の論理時刻 `t`（ms）を含む（MUST、[06](./06-data-models.md)）。
- 受信側は `推定相手時刻 = ローカル時刻 + offset` を用いて、状態を現在時刻へ外挿してから描画する（SHOULD）。

## 5.3 リモートエンティティの補間/外挿

- リモートプレイヤー位置: 直近2〜3サンプルから線形補間し、欠落時は速度で外挿（SHOULD）。
- 大きな乖離（しきい値超）が出たら**スナップ補正**しつつ、見た目は短時間でブレンドする（SHOULD）。
- **MUST NOT**: リモート状態の急なワープがプレイを破綻させないこと（補正は視覚的に滑らかに）。

## 5.4 ボール権限とハンドオフ（最重要）

### 5.4.1 権限の所在

- ボールが**自コート**（[02](./02-coordinate-and-camera.md) 自半面）にある間、そのピアが**ボール権限**を持つ（MUST）。
- 権限保持ピアは:
  - 自端末でボール物理・ヒット判定・バウンド/イン-アウト判定を権威的に計算する（MUST）。
  - BALL_STATE を高頻度で相手へ送る（MUST）。
- 非権限ピアは受信したボール状態を補間表示し、**ヒット判定を行わない**（自コートに来るまで）。

### 5.4.2 ヒットとハンドオフのシーケンス

```
[権限: Player X（自コート）]
  1. X がボールに対しA/Bを入力 → ローカルで即ヒット成立（体感遅延0）
  2. X はヒット後ボール状態（位置/速度/高さ/スピン/種別/hitId/t）を確定
  3. X が BALL_HANDOFF(reliable) を相手 Y へ送信
  4. ボールがネット (NET_Y) を越え Y のコートへ入った時点で
     権限は X → Y へ移譲（ハンドオフ確定）
  5. Y は受領したボール状態から自端末で物理継続し、Y のヒットを待つ
  6. 以降 5.4.1〜を Y 視点で繰り返す
```

要件:
- **MUST**: ヒットはローカルで即時成立し、ネットワーク往復を待たない（空振りゼロ保証）。
- **MUST**: BALL_HANDOFF は reliable/ordered で送る。取りこぼすとボールが消失/二重化する。
- **MUST**: 各ハンドオフに単調増加の `hitId`（または `seq`）を含め、受信側は古い/重複ハンドオフを破棄する。
- **MUST**: 権限の二重保持（両者が同時に権威的計算）を避ける。ネット越え＝権限移譲を唯一の移譲トリガとする。

### 5.4.3 権限の重なり・競合の解決

ネット付近やラグで「どちらの自コートか」が曖昧になるケースの規則:
- **MUST**: ボールの論理 `y` が `NET_Y` を越えた瞬間を移譲点とする（送り手は越えた時点で権限を放棄）。
- **SHOULD**: 移譲直後の短いグレース期間（例 100ms）は、受け手が古い自分発の状態より新しいハンドオフを優先する。
- **MUST**: 双方が同時にヒットを主張した（同一ボールに2つのhitId）場合、**ホストが裁定**: より小さい`t`（先に打った方）を採用し、他方を破棄。結果を全員へ通知（[04](./04-game-rules.md)）。

### 5.4.4 得点確定

- **MUST**: イン/アウト/ネット/2バウンド/取り逃しを検知したボール権限ピアが、`POINT_REPORT` をホストへ送る。
- **MUST**: ホストは報告を検証（競合裁定含む）し、`POINT_RESULT`（得点者・新スコア・次サーブ）を全員へ配信する。これが唯一の正。
- **MUST**: クライアントは自分のローカル判定よりホストの POINT_RESULT を優先する。

## 5.5 ダブルスの同期

- **SHOULD**: 自チーム半面のボール権限は「ボールに最も近い/最後に触れた味方」が代表として持つ。
- **MUST**: チーム内代表の決定ロジックを定め、二重ヒットを防ぐ（例: より近いプレイヤーがヒット権、僅差はホスト裁定）。
- ホスト経由で相手チームへ中継される（[01](./01-architecture.md) スター型）。中継遅延を最小化すること（SHOULD）。

## 5.6 接続・切断・再接続

| 事象 | 要件 |
|---|---|
| 接続待ち | ロビーで参加状況を表示。定員到達でホストが開始可能（MUST）。 |
| 一時的パケット欠落 | unreliable状態は欠落許容。重要イベントはreliableで保証（MUST）。 |
| クライアント切断 | ホストが検知し、シングルスは試合中断/終了、ダブルスは「相手1名離脱」を通知（MUST）。 |
| ホスト切断 | ルーム消滅。全クライアントに通知し、トップへ戻す（MUST）。 |
| 再接続 | 同 roomId への再参加を試行してよい（MAY）。状態復元は best-effort。 |
| タイムアウト | 一定時間（例 10s）応答なしで切断扱い（SHOULD）。 |

## 5.7 反チート方針（軽量）

カジュアル前提のため強固な対策は範囲外（[00](./00-overview.md)）。ただし:
- **SHOULD**: ホストは明らかに不正な値（コート外への瞬間移動、非現実的速度）をサニティチェックで拒否/クランプする。
- **MUST NOT**: 1クライアントの不正報告だけでスコアが確定しないこと（ホスト裁定を経る）。

## 5.8 検証観点（→ [07](./07-acceptance-criteria.md)）

- 人工遅延（例 150ms）注入下でも、自コートのスイングが体感遅延なく当たる。
- ハンドオフ往復でボールが消えない/二重化しない（hitId重複破棄が機能）。
- 同時ヒット競合時にホスト裁定で1つに収束する。
