# 02. 座標系・画面180度反転・カメラ制御

## 2.1 設計原則

全プレイヤーが「**手前（画面下側）が自分**」という視点で快適にプレイできること。
そのために、**全ピア共通の論理座標（Canonical Space）** と、**各端末固有のローカル描画座標** を明確に分離する（MUST）。

> 同期は常に論理座標で行う。反転は描画と入力の最終段でのみ行う（MUST）。
> これにより「物理・通信は1つの真実、見え方だけ各自で変換」が保たれ、反転起因のズレを防ぐ。

## 2.2 論理座標（Canonical Space）

- 基準は**ホスト視点**。原点は左上、x:右が正、y:下が正（Phaserの標準と一致）。
- コート寸法を論理単位で定義する（実装定数、例として下表）。

| 定数 | 既定値（論理単位） | 説明 |
|---|---|---|
| `COURT_W` | 800 | コート横幅 |
| `COURT_H` | 1200 | コート縦（奥行き）。比率は16:9想定の描画にスケール |
| `NET_Y` | 600（= COURT_H/2） | ネットのy座標 |
| ホスト側コート | `y > NET_Y`（下半分） | ホスト（A）が手前 |
| 相手側コート | `y < NET_Y`（上半分） | 相手が奥 |

- ボール・全プレイヤーの位置/速度は**論理座標で保持・送信**する（MUST）。

## 2.3 ローカル描画変換

各ピアは論理座標 `(x, y)` をローカル描画座標 `(lx, ly)` に変換してから描画する。

### ホスト（プレイヤーA）= 恒等変換

```
lx = x
ly = y
```

### クライアント（プレイヤーB）= 180度回転

```
lx = COURT_W - x
ly = COURT_H - y
```

- 速度ベクトルも同様に変換する: `lvx = -vx, lvy = -vy`（クライアント）。
- **MUST**: 描画・カメラのみに変換を適用し、論理状態（同期対象）は変換しないこと。
- **MUST**: クライアントの画面でも、自キャラが下半分・相手が上半分に見えること。

### ダブルスの座標割り当て

- ホスト側チーム（A, C）: 論理下半分。クライアント側チーム（B, D）: 論理上半分。
- 各プレイヤーは「自分が下に見える」変換を個別に適用する。
  - ホスト陣営（A,C）は恒等、相手陣営（B,D）は180°反転。
- 左右の相方の位置関係も反転に追従すること（MUST）。

## 2.4 カメラ・レイアウト

- **MUST**: 横画面（ランドスケープ）固定。縦持ち時は「横にしてください」案内を表示（[03](./03-controls-ux.md)）。
- **SHOULD**: コート全体が常に画面に収まるフィット表示（プレイヤーがコート外の死角に入らない）。
- **MAY**: 自キャラ寄りに僅かにカメラオフセットしてもよいが、相手・ボールが画面外に出ないこと。
- スコア表示は常にローカル画面の上部に、各自視点で「自分（自チーム）のスコアが分かる」よう表示する（MUST、[04](./04-game-rules.md)）。

## 2.5 入力方向の反転同期

- 入力（移動・打ち分け方向）は**ローカル座標系での操作**としてユーザーに提示する（右に倒せば画面の右）。
- それを論理座標へ逆変換してから物理・同期に渡す（MUST）。
  - ホスト: 恒等。
  - クライアント: 入力ベクトル `(ix, iy)` → 論理 `(-ix, -iy)`。
- 結果として、両プレイヤーとも「自分から見て右へ動かせば自分のキャラが右へ動く」体験になること（MUST）。打ち分け方向（[03](./03-controls-ux.md) 4.2）も同じ規則で論理化する。

## 2.6 検証観点（→ [07](./07-acceptance-criteria.md)）

- 2台で接続し、片方を180°反転表示しても、両画面で「自分が手前」かつ位置が破綻なく同期すること。
- クライアントで右入力 → 相手（ホスト）画面では左方向への移動として正しく映ること（同一の論理移動を双方が一致して描画）。
